中间件简述

Concept of Middleware

Concept: Middleware sits in the “middle” to allow communication and interaction among heterogeneously different applications and platforms.

Systems of engagement <– Middleware –> Systems of Record

The Layer of Middleware: 1. Application Layer 2. Middle Layer (Middleware) 3. Platform Layer

Types of Middleware

面向对象中间件 (Object-Oriented Middleware, OOM)
这种中间件是基于面向对象的原则设计的，支持对象的创建、管理和通信。在分布式系统中，它可以帮助不同的组件或应用程序之间的交互。

  • **远程方法调用 (Remote Method Invocation, RMI)**：允许应用程序或系统在远程之间进行同步和异步的交互。换句话说，一个应用程序可以在另一台计算机上的另一个应用程序中调用方法，而不必担心通信的底层细节。

  • **远程过程调用 (Remote Procedure Call, RPC)**：是客户端与服务器之间通信的一种方法，客户端请求服务，服务器提供服务。

  • **对象请求代理 (Object Request Broker, ORB)**：允许一个计算机上的程序通过计算机网络调用另一台计算机上的程序。例如，共同对象请求代理架构 (Common Object Request Broker Architecture, CORBA) 是一种广泛使用的ORB规范，允许跨多种编程语言和平台进行通信。

**事务处理监视器 (Transaction Processing Monitor, TPM)**：用于执行和强制执行电子事务。在商业应用中，它确保所有数据库操作都能完成或者都不会发生，以维护数据的完整性和一致性。

**反射中间件 (Reflective Middleware, RM)**：能够自我操作和推理。这意味着它可以自适应地更改其行为或配置，以响应外部条件或内部状态。

**消息导向中间件 (Message-Oriented Middleware, MOM)**：使用消息作为通信方法。与直接的远程方法调用或过程调用相比，消息导向中间件允许更高的解耦、异步通信和更好的伸缩性。

1 Message-oriented Middleware (MOM)

Messaging

Messaging

一种软件组件或应用程序之间的通信方法，具有peer-to-peer能力，采用loosely coupled的分布式通信方式（也就是说，消息的发送者和接收者不必同时在线或运行来进行交互）。在消息传递系统中，每个客户端都连接到一个消息代理。这个消息代理为创建、发送、接收和阅读消息提供了必要的设施。

Messaging Services

消息服务是允许用户实现Messaging的软件组件功能。在此系统中，每个组件都可以既充当客户端又充当服务器，这样使得组件之间的通信更加灵活和动态。

Advantages of Messaging

  1. Heterogeneous (multiple) Integration （异构集成）
  2. Reduce System Bottlenecks （减少系统瓶颈）
  3. Increase Scalability （增加可伸缩性）

Type Messaging Services

  1. Cloud Messaging Service: Using distributed heterogeneous systems and Message-Oriented Middleware (MOM)
  2. Short Message Service (SMS) （短信服务，不重要）
  3. Instant Message Service (IMS) （即时消息服务，不重要）
  4. Multimedia Message Service (MMS) （彩信服务，不重要）

Message-oriented Middleware (MOM)

Message-oriented Middleware (MOM)
面向消息的中间件 (MOM) 是一种软件或硬件，通过在通信系统之间传递消息，为异构的应用程序、平台和系统提供通信渠道。

MOM Architecture

MOM Client - Producer

客户端使用“MessageProducer”对象来生成消息，消息随后由消息生产者发送到在API中使用目的地对象写入的物理目的地。

Message Broker

代理负责接收和发送消息，并维护一个持久的数据存储，以可靠地发送数据。同时代理使用加密的SSL认证连接提供安全连接，并监控消息和连接活动，为管理员提供故障排除期间访问的诊断信息日志。

MOM Client - Consumer

MOM使用异步消费方法，消息自动传递给由消费者定义的“MessageListener”对象。或者，只有与消费者选择的属性匹配的消息可以被消费。

Types of MOM

  • Message Queuing (P2P): 在这种模型中，发送者与接收者之间存在明确的一对一关系。这意味着，如果发送者想要发送消息，他必须知道接收者的确切地址或标识。此外，接收者需要知道有消息正在向其发送。这种模型常用于需要明确的消息传递路径和有明确目标的情境。

  • Types of Queuing Systems in MOM:

    **先进先出 (FIFO)**：这是最基本的队列策略。在这种策略中，消息的接收顺序与其发送顺序相同。

    具有优先级的FIFO：虽然基于FIFO的原理，但它允许消息基于优先级进行排序。例如，高优先级的消息可能会在低优先级的消息之前被处理。

    日志队列：这是一个特殊的队列，系统会跟踪并保留所有消息的副本。这对于确保数据完整性和恢复功能非常有用。

    公共队列：这种队列对所有用户都是开放的，意味着任何用户都可以发送或接收消息。

    私有队列：与公共队列相反，私有队列需要用户进行身份验证和授权才能使用。

    连接器/桥接器：这不是传统意义上的队列，而是一个代理或中间件，用于连接不同的MOM实现或队列，尤其是商业MOM。

  • Publish/Subscribe: 在这种模型中，发送者与接收者之间存在一对多或多对多的关系，发送者（或发布者）只需发布他们的消息，而不需要知道谁会接收它。接收者（或订阅者）可以订阅他们感兴趣的消息，而不需要知道是谁发布的。这种解耦方式使得Pub/Sub模型非常适合于分布式和动态的环境，如云系统，因为组件可以自由地进入和离开系统，而不会影响其他组件。

  • Types of Publish-Subscribe Message:

    Topics-Based（基于主题的）：在这种方法中，发布者将消息发布到特定的主题，而订阅者则订阅他们感兴趣的主题。主题通常是预先定义的，并使用特定的术语（如政治、数学、科学等）进行分类。这种方法的缺点是它可能太受限，因为主题是固定的，并使用特定的术语。Hierarchical Topics, Independent Channels

    Content-Based（基于内容的）：与基于主题的模型不同，基于内容的模型不依赖于预先定义的主题。相反，订阅者可以使用查询语法来指定他们感兴趣的消息内容。这提供了更大的灵活性，因为订阅者不受到固定主题的限制。**(Un)structured queries**, Complex Event Processing

MOM Implementation Models

  • Microsoft’s Message Queuing (MSMQ)

  • Oracle’s Java Messaging Service (JMS)

  • IBM’s WebSphere Message Queuing (WebSphere MQ)

  • Sonic Message Queuing (Sonic MQ)

Advantages of MOM

  • Hides the complexity of applications running on different operating systems （隐藏复杂性）
  • The mode of sending and receiving messages is reliable （可靠性）
  • MOM embeds security features so that messages cannot be intercepted and changed （安全性）
  • It sends messages asynchronously and so the receiver does not have to know that messages are being sent prior to sending them （异步通信）
  • Allows heterogeneous systems to communicate （异构系统通信）
  • Allows different programming languages to be ported (portability) across different platforms （可移植性）

Security in MOMs

• There are implementation of access controls between the client and the API for sending and receiving messages • IBM uses System Authorisation Facility (SAF) to control access to MOM • Encrypted Messages • The Qpid Advanced Message Queuing Protocol (AMQP) uses SASL/TLS (Simple Authentication and Security Layer/Transport Layer Security) for authentication and encryption of messages in Java Messaging Service (JMS)

2. Java Message Service (JMS)

Introduction

JMS是一个Java API，允许应用程序创建、发送、接收和读取消息。支持loosely coupled communication和Asynchronous communication，并确保消息只被传递一次。
官方文档：[基本 JMS API 概念 - Java EE 6 教程 (oracle.com)](https://docs.oracle.com/javaee/6/tutorial/doc/bncdx.html#:~:text=JMS API Architecture 1 A JMS provider is,objects that communicate information between JMS clients. 更多项目)

JMS API Architecture

JMS Provider：实现了JMS接口的消息中间件的供应商，如：Apache ActiveMQ, IBM MQ, RabbitMQ(虽然主要是AMQP, 它也支持JMS) 。

JMS Clients: 是使用 Java Message Service (JMS) API 的应用程序或组件。这些客户端可以创建、发送、接收和读取消息，而不必关心消息传输的底层细节。简单来说，JMS Clients 是与 JMS Providers 交互的那些程序。分为一下两种类型：

  1. 生产者 (Producers): 生产者是创建并发送消息到目的地（例如队列或主题）的客户端。生产者使用 JMS API 提供的方法创建消息，并将其发送到指定的目的地。
  2. 消费者 (Consumers): 消费者是从目的地（例如队列或主题）接收消息的客户端。消费者可以是监听特定目的地的消息的程序，或者它可以主动地从队列中提取消息。

根据 JMS 的两种消息传递模型，即点对点模型 (P2P) 和发布/订阅模型 (Pub/Sub)，消费者的行为也有所不同：

  • 在 点对点模型 中，消息被发送到队列，由一个消费者接收。一旦被消费者接收，该消息就不再存在于队列中。
  • 在 发布/订阅模型 中，消息被发送到主题，并且可以被多个订阅该主题的消费者接收。消息会传递给所有当前活动的订阅者。

Messages: These are the objects that communicate information between JMS Clients.

Administered Objects: 在 JMS 架构中，Administered Objects 是那些由系统或应用管理员预先配置的对象。不同的JMS Provider可能有不同的管理和配置要求，Administered Objects为 JMS 客户端提供了一个方便的方式来与消息服务交互，而不需要每次都进行复杂的配置。下方是两种主要的 Administered Objects:

  1. Destinations (D): 这是消息的最终地址或来源。在 JMS 中，有两种类型的目的地：队列（用于点对点消息传递）和主题（用于发布/订阅消息传递）。
  2. Connection Factories (CF): 这些对象知道如何连接到特定的 JMS Provider。当 JMS 客户端需要与消息服务交互时，它会使用 Connection Factory 创建一个连接。

JNDI Namespace：Java Naming and Directory Interface (JNDI) 是一个Java API，它允许Java应用程序在目录中查找和访问对象及数据。在JNDI Namespace中的数据（常为对象的命名和绑定引用）一般来源于应用程序或应用服务器的配置设置。

当Java应用程序或JMS客户端需要与JMS Provider建立连接以发送或接收消息时，它们通常会通过查询JNDI Namespace中为这些对象分配的名称来获取Administered Objects的定义。

Java Messaging Domains

  1. Point-to-point (P2P) messaging domain:

    消息从Message Producer发送到特定队列，然后Message Consumer从中提取消息，每条消息在队列中只能被一个消费者消费。但是，多个消费者可以侦听同一队列，此时谁先到谁先得。如果没有消费者可用或消息尚未被消费，队列会保持消息，直到有消费者可以处理它或它到达设定的过期时间。

  2. Publish/Subscribe messaging domain:

    这是基于主题的传递模式，与公告板或广播类似。Publishers和subscribers是匿名的，并且可以动态地发布订阅。 每条消息都可以有多个客户端。出版者和订阅者之间存在时间依赖性。为一个主题订阅的客户端只能消费在客户端创建订阅之后发布的消息。

Types of JMS Message Consumption

Synchronous: 订阅者或接收者通过调用receive方法明确地从目标获取消息。receive方法可以阻塞，直到消息到达，或者如果消息在指定的时间限制内未到达则超时。

Asynchronous: 客户端可以为消费者注册一个消息监听器，每当消息到达目标时，JMS提供者通过调用监听器的onMessage方法来传递消息，该方法会对消息的内容进行操作。这种方法允许消费者在等待消息时进行其他操作，因为当消息到达时，它会自动得到通知。

The JMS API Programming Model

Create Connection Factories (CF)

JMS有两种连接工厂QueueConnectionFactory 和 TopicConnectionFactory，分别对应队列（Point-to-Point）和主题（Publish/Subscribe）。CF作为Administered Objects，通常会通过查询JNDI Namespace来获取定义。

Create Connections

JMS有两种连接QueueConnection和TopicConnection，分别可由QueueConnectionFactory 和 TopicConnectionFactory进行创建。

Create Destinations

JMS有两种目标地址Queue 和 Topic，分别对应队列（Point-to-Point）和主题（Publish/Subscribe）。Destinations作为Administered Objects，通常会通过查询JNDI Namespace来获取定义。

Create Sessions

会话提供了一个用于发送和接收消息的序列化环境。这意味着在一个会话中的所有操作，比如发送和接收消息，都是串行的，不会并发执行。JMS有两种会话QueueSession和TopicSession，分别可由QueueConnection 和 TopicConnection进行创建。

Create Message Producers

JMS有两种Producers QueueSender和TopicPublisher分别可由QueueSession和TopicSession进行创建。

Sent Message

一旦消息被创建，你可以使用之前创建的QueueSender或TopicPublisher来发送消息。

Create Message Consumers

JMS有两种ConsumerQueueReceiver和TopicSubscriber分别可由QueueSession和TopicSession进行创建。

Receive Message

可以使用创建的QueueReceiver或TopicSubscriber使用receive方法来消费消息。

收到的消息可以使用getter方法将内容提取出来，如：getText可将消息转换为字符串。

Message Listeners

实现了MessageListener接口，该接口包含一个名为onMessage的方法，在onMessage方法中，你定义消息到达时要采取的操作。

Message Selectors

JMS提供了一种功能，允许消息消费者基于某些条件筛选他们想要接收的消息。例如，一个应用可能只对具有特定属性或值的消息感兴趣。使用消息选择器，这个应用可以确保它只会接收满足这些条件的消息。
